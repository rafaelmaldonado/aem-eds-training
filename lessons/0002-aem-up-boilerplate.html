Stand Up the Boilerplate & Run It Locally

Your first EDS project — no Maven, no archetype, no local AEM instance.

In Lesson 1 you learned the doc→DOM→decorate() pipeline. Now you'll turn that knowledge into a skill: get a real project running on your machine. The whole goal of this lesson is to see localhost:3000 serve an EDS page you can edit live.

The unlearning: There's no local AEM server. aem up is a tiny proxy — it serves your local blocks/, styles/, and scripts/, but fetches the actual content (the rendered DOM) from your .aem.page preview in the cloud. Code is local; content is remote.

The five steps

All quoted from the official aem.live developer tutorial.

  1. Create your repo from the template. Open github.com/adobe/aem-boilerplate, click Use this template → Create a new repository. Make it public.
  2. Install the AEM Code Sync GitHub app on that repo (install link) — choose Only select repositories and pick your new one. This is what makes git push deploy. Your site now lives at: 
    https://<branch>--<repo>--<owner>.aem.page/
  3. Install the CLI and clone: 
    npm install -g @adobe/aem-cli
git clone https://github.com/<owner>/<repo>
  4. Start the local dev server: 
    cd <repo>
aem up
    This opens http://localhost:3000/.
  5. Edit and watch. Change any file under blocks/ or styles/ and the browser reflects it immediately. To deploy: add, commit, push — code sync updates preview (.aem.page) and production (.aem.live).
Where does the content come from? The boilerplate ships wired to a demo content source. aem up pulls page content from your preview and overlays your local code on top — so you develop code offline while authoring happens in Docs/DA. We'll cover connecting your own content source and the preview/publish model in a later lesson.
