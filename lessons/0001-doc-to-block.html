From a Doc Table to a Decorated Block

The one pipeline that replaces components, dialogs, and HTL.

You already know the classic AEM loop: a component is a Java/Sling model + an HTL template + a dialog, and the server renders it from the JCR. Edge Delivery throws all of that out. There is no repository, no dialog, no server-side rendering of your logic. Instead, a single pipeline turns a document into HTML and then lets your JavaScript enhance it in the browser.

This lesson teaches that pipeline end to end. It's the mental model everything else in EDS hangs off of.

The unlearning: In classic AEM the author fills a dialog and the server renders your component. In EDS the author writes a table in a Google Doc, and the browser runs your decorate() function. Same goal — reusable authored content — opposite machinery.

The four stages

Every block travels the same road:

  1. Author writes a table in a document. First row = the block name.
  2. The pipeline converts that table to clean, semantic <div> markup — one div per row, one per cell.
  3. The block loader (in scripts.js) wraps it, adds CSS classes and data attributes, and lazy-loads the block's code.
  4. Your decorate(block) function reshapes that DOM into whatever the design needs; your CSS styles it.

Stage 1 → 2: the table becomes DOM

An author writes a two-row table whose header cell says Hero. That single-cell block is delivered as this markup (source):

<div class="hero">
  <div>              <!-- a row -->
    <div>            <!-- a cell -->
      <p>Hello, World.</p>
    </div>
  </div>
</div>

Note what the block name did: it became the folder (blocks/hero/), the filenames (hero.js, hero.css), and the CSS class on the wrapper. One name, three jobs.

Block options are authored in parentheses: Hero (dark, wide) → class="hero dark wide". That's your EDS equivalent of dialog fields that toggle a style variant — no dialog required.

Stage 3 → 4: decorate() reshapes it

Each block is an ES module at blocks/hero/hero.js exporting a default function. The loader calls it with the block element:

// blocks/hero/hero.js
export default function decorate(block) {
  // `block` is the <div class="hero"> above.
  // Reshape the raw row/cell divs into the DOM your CSS wants.
  const picture = block.querySelector('picture');
  const text = block.querySelector('p:not(:has(picture))');
  block.replaceChildren(picture, text); // flatten the wrapper divs
}

This is your render logic — but it runs in the browser, after semantic HTML already exists. That's why EDS pages are fast and SEO-friendly even before your JS runs: stage 2 already shipped meaningful markup.
