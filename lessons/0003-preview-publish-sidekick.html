Content, Preview & Publish with Sidekick

Where the content lives, and the two-step road from a doc to the live site.

In Lesson 2 you ran aem up and saw it serve local code over remote content. This lesson answers the obvious next question: where is that content, and how does an edit become a live page? In classic AEM you knew this cold — author, activate, dispatcher. EDS keeps the two-step shape but changes every piece.

The unlearning: There's no JCR and no dispatcher flush. Content is a document (Google Docs / SharePoint / DA). You Preview it to .aem.page (a staging URL), then Publish it to .aem.live (production). Sidekick is the little toolbar that fires both — your author/activate buttons, living in the browser.

What connects a doc to your project?

A single config file in your repo, fstab.yaml, maps your site to its content source — the Drive folder, SharePoint site, or DA path. That mapping is why the pipeline knows which document backs which URL (source).

# fstab.yaml — mountpoints: URL path → content source
mountpoints:
  /: https://drive.google.com/drive/folders/<folder-id>

The Preview → Publish loop

  1. Install Sidekick — the AEM Sidekick browser extension — and add your project (its bookmarklet/config points at your repo).
  2. Edit the document in Docs/DA. Nothing is live yet; the doc is just a doc.
  3. Preview. Open the doc, click Preview in Sidekick. The page renders at <branch>--<repo>--<owner>.aem.page/<path>. This is also the content aem up overlays your local code onto.
  4. Publish. Happy with it? Click Publish. The page goes live at the matching .aem.live URL and the CDN is updated.
Code vs. content — two independent pipelines. git push deploys code (from Lesson 2). Preview/Publish moves content. They meet only at render time: published content + deployed code = the live page. Changing a block's JS never requires re-publishing a doc, and vice versa.

Install Sidekick, open the boilerplate's demo doc (or your own), and run one full loop: edit → Preview → check .aem.page → Publish → check .aem.live. The primary source walks the authoring flow:
